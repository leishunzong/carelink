# 城市编码设计说明

## 📍 城市因素设计

护联系统采用**单城市绑定**模式，用户和护工都绑定唯一的服务城市，只在本城市内提供/使用服务。

## 🗄️ 数据库设计

### user表
```sql
city_code   VARCHAR(10)  NULL COMMENT '城市编码（如110100北京市）'
city_name   VARCHAR(50)  NULL COMMENT '城市名称'
```

- **可为空**：用户登录后可以暂不设置城市
- **首次使用**：下单前需要设置城市

### caregiver表
```sql
city_code   VARCHAR(10)  NOT NULL COMMENT '服务城市编码'
city_name   VARCHAR(50)  NOT NULL COMMENT '服务城市名称'
```

- **必填**：护工注册时必须选择服务城市
- **固定服务**：只能在该城市接单

## 🔑 Redis GEO 存储结构

### 按城市分区存储

```
旧设计：
  caregiver:locations:available

新设计：
  caregiver:locations:available:110100  (北京市的接单护工)
  caregiver:locations:available:310100  (上海市的接单护工)
  caregiver:locations:available:440100  (广州市的接单护工)
```

### 优势
- ✅ 自动同城隔离
- ✅ 查询效率高（只查本城市）
- ✅ 数据量小（每个城市独立）
- ✅ 易于扩展（新城市自动创建新键）

## 🔄 业务流程

### 用户端流程

```
1. 用户注册登录
   ↓
2. 前端定位获取城市（如：北京市，110100）
   ↓
3. 调用接口设置城市
   POST /api/user/city
   { "cityCode": "110100", "cityName": "北京市" }
   ↓
4. 用户下单时自动使用用户的cityCode
   → 只从北京市的护工池中匹配
```

### 护工端流程

```
1. 护工注册（必须选择服务城市）
   POST /api/caregiver/register
   {
     "username": "hg001",
     "password": "123456",
     "phone": "13900139001",
     "realName": "刘阿姨",
     "gender": 2,
     "cityCode": "110100",
     "cityName": "北京市"
   }
   ↓
2. 护工上报位置
   POST /api/caregiver/location
   longitude=116.404&latitude=39.915
   ↓
3. 护工切换为接单中
   POST /api/caregiver/work-state?workState=1
   → 添加到 caregiver:locations:available:110100
   ↓
4. 用户下单（北京市）
   → 从 caregiver:locations:available:110100 中查找附近护工
```

## 🎯 派单匹配逻辑

### 搜索同城护工
```java
// 用户下单时
User user = userService.getById(userId);
String cityCode = user.getCityCode(); // 110100

// 搜索该城市附近5km的护工
List<Long> nearbyIds = caregiverLocationService.searchNearbyCaregivers(
    cityCode,  // 城市编码
    116.404,   // 用户位置经度
    39.915,    // 用户位置纬度
    5.0,       // 5公里
    10L        // 最多10个
);

// 返回：北京市范围内，距离用户5km内的护工ID列表
```

### Redis命令查看

```bash
# 查看北京市所有接单中的护工
ZRANGE caregiver:locations:available:110100 0 -1

# 搜索北京市指定位置附近的护工
GEORADIUS caregiver:locations:available:110100 116.404 39.915 5 km WITHDIST ASC

# 查看上海市的护工
ZRANGE caregiver:locations:available:310100 0 -1
```

## 🔐 安全性和数据隔离

### 自动隔离
- 北京的用户只能匹配北京的护工
- 上海的护工不会出现在北京的派单中
- 每个城市的Redis GEO独立

### 防止跨城服务
- 用户下单时检查 `user.cityCode` 是否为空
- 搜索护工时使用用户的 `cityCode`
- 护工只能接到本城市的订单

## ⚙️ 数据维护

### 护工更换服务城市
```java
// 需要做的事：
// 1. 从旧城市的Redis GEO删除
// 2. 更新数据库城市信息
// 3. 如果是接单中状态，添加到新城市的Redis GEO
```

**建议**：护工端暂不提供更换城市接口，避免频繁切换。如需更换，联系管理员。

### 用户更换城市
```java
// 用户可以随时更换城市
POST /api/user/city
{ "cityCode": "310100", "cityName": "上海市" }

// 影响：
// - 之后的订单在新城市匹配护工
// - 历史数据不受影响
```

## 🎁 总结

**核心优势**：
1. **同城匹配**：自动保证同城服务
2. **性能优异**：Redis按城市分区，查询速度快
3. **数据隔离**：城市之间数据完全隔离
4. **易于扩展**：新增城市自动支持
5. **符合业务**：居家养老本就是本地化服务

现在系统已支持多城市运营！🎉
